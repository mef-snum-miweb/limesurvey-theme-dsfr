# Revue complète du thème — 9 juin 2026

> Revue en profondeur du thème (v1.4.0) : architecture JavaScript, CSS, templates Twig, iso-fonctionnalité avec le thème vanilla de LimeSurvey 6, conformité DSFR 1.14 et couverture des fonctionnalités du backoffice.
>
> **Méthode** : 5 analyses parallèles (JS / CSS / types de questions vs core / layouts & options / conformité DSFR), avec pour référence le core LimeSurvey master du 2026-06-09 (commit `d1a569c`) — les vues `application/views/survey/questions/answer/` dont héritent tous les thèmes. Le bundle `scripts/custom.js` a été reconstruit et diffé : **strictement à jour** par rapport à `src/`.
>
> **Suivi** : le plan d'action est découpé en épics et issues GitHub (labels `P0-critique`, `P1-iso-fonctionnalite`, `P2-dette`).

## Verdict global

Thème de **bonne facture générale** : ~75-80 % de conformité structurelle DSFR, pipeline de build exemplaire (`src/` modulaire + `build:theme:check` en CI), documentation in-code remarquable avec références RGAA. Mais **9 constats critiques** et une trentaine de majeurs, autour de trois dettes structurelles :

1. `scripts/theme.js` legacy qui **contredit activement** les modules `src/` ;
2. des **variables CSS DSFR fantômes** qui annulent silencieusement les styles d'erreur ;
3. des **écarts d'iso-fonctionnalité réels** sur dual-scale, sliders, dates et plusieurs attributs backoffice.

### ⚠️ Avertissement de version

Le master LimeSurvey contient des renommages récents de champs POST et d'ids (`{name}_Cother`, `{fileid}_Cfilecount`, `javatbd{name}_CY`, `javatbd{name}_S{aid}`…). Le thème suit l'ancienne convention (`{name}other`, `_filecount`, `javatbd{name}Y`…). L'image déployée étant `martialblog/limesurvey:6-apache` (suit la dernière 6.x), **chaque montée de version peut transformer ces écarts en pertes de réponses silencieuses**. À vérifier contre la version réellement déployée, et figer la version LS testée.

## 🔴 Constats critiques

### JavaScript

**C1 — `scripts/theme.js:1415-1444` sabote la validation DSFR de `src/`.** Un handler jQuery délégué (`input change keyup`) retire `fr-input-group--error` dès la première frappe — même si la saisie reste invalide (posée par `numeric-validation.js`) — et son test « plus d'erreurs » ne regarde que `.ls-question-mandatory*`, jamais les `.fr-message--error` : `updateErrorSummary()` considère la question corrigée à tort. → Supprimer ce bloc legacy.

**C2 — `:has()` non protégé peut tuer toute l'initialisation.** `src/validation/required-fields.js:21` lève une `SyntaxError` sur Firefox < 121 (dont **ESR 115**) et Safari < 15.4. Appelé en 7ᵉ position de `onReady` (`src/index.js:62`), **avant** errors-dsfr, error-summary, combobox, ranking… → une exception interrompt tout le reste. → try/catch + sandboxer chaque init de `index.js`.

### CSS

**C3 — 10 variables CSS inexistantes dans DSFR 1.14.4** : `--error-425` (custom.css:411,419,485,491), `--warning-425`, `--success-425`, `--info-425`, `--blue-france` (theme.css:304,1281,1290), `--grey-light`, `--border-action-high-blue-france-hover`, `--focus-blue-france`, `--border-radius`. Une déclaration `color: var(--inexistante)` sans fallback tombe en `unset` — et le `!important` de custom.css:411 **gagne la cascade puis s'annule**, écrasant la règle valide de theme.css:974. Concret : **les messages d'erreur EM (`.ls-em-error`) ne sont pas rouges**. Bug fonctionnel + RGAA. → Remplacer par `--error-425-625` etc. (~15 lignes).

### Iso-fonctionnalité

**C4 — Dual-scale (type `1`, no_dropdown) : option « Sans réponse » supprimée.** `views/survey/questions/answer/arrays/dualscale/answer.twig:169-171` laisse la cellule vide (`{# handled by JavaScript #}`), mais **aucun JS du thème ne la gère** (grep négatif). Question non obligatoire avec `shownoanswer` → réponse forcée, donnée faussée.

**C5 — Dual-scale : ids construits avec `#`.** `answer{{myfname0}}` où `myfname0` = `{SGQA}#0`, alors que le core utilise `myfid0` (`{SGQA}_0`) pour tous les ids. Les sélecteurs jQuery d'ExpressionManager échouent → **relevance, validation, équations et tailoring cassés** sur toute question dual-scale. (La variante dropdown `answer_dropdown.twig` est correcte.) Aussi : hidden fields enfants directs de `<tbody>` (HTML invalide, l.182-192).

**C6 — Sliders multiplenumeric (type `K`) : sémantique « non touché » perdue.** Le `<input type=range>` natif a toujours une valeur → **une question slider jamais touchée soumet une réponse**. `slider_accuracy|default('1')` référence une variable inexistante (le core passe `slider_step`) → pas toujours = 1 ; `slider_default`, `slider_handle`, `prefix/suffix`, `separator` perdus ; le reset remet `(min+max)/2` au lieu de l'état vide. (`rows/sliders/horizontal_slider.twig:8-19`, `vertical_slider.twig:16-30`.)

**C7 — Date (type `D`, mode selector) : `<input type=date>` soumet de l'ISO** quel que soit `dateformatdetails` du questionnaire (le serveur parse au format d'affichage). Les formats datetime ne sont plus saisissables ; pré-remplissage converti uniquement pour `DD/MM/YYYY` ; `min/max` sans `processString` (expressions EM dans date_min/date_max non évaluées). (`date/selector/answer.twig:37-82`.)

### Layouts / DSFR

**C8 — `.ls-js-hidden` n'est défini nulle part.** `config.xml:43` fait `<remove>template-core.css</remove>` (qui définissait la classe) sans la redéfinir. Concret : l'alerte d'erreur RGPD `#datasecurity_error` (`privacy_text.twig:22`, `privacy_modal.twig:22`) **s'affiche en permanence** dès que la politique de sécurité des données est activée. Touche aussi `date/dropdown/answer.twig:28`, `layout_survey_list.twig:47`, `register_form.twig:25`. → `.ls-js-hidden{display:none!important}` dans custom.css.

**C9 — `justsaved.twig` 100 % Bootstrap** (`alert alert-success fade in`, classe `fr-alert--dismissible` et API `data-fr-dismiss` inventées), inclus sur **toutes** les pages via `mainrow.twig:25`. La non-conformité DSFR la plus visible.

## 🟠 Constats majeurs

### JavaScript (`src/`, `scripts/`)

- **Compteur d'erreurs jamais ré-inséré après `remove()`** — `mst-errors.js:149-174`, `array-validation.js:145-169` : si l'utilisateur re-vide un champ, le conteneur est détaché du DOM → question en erreur sans message visible ni annonce SR. Le chemin `handleChoiceArray` (array-validation.js:249) est, lui, corrigé via `isConnected`. *Lié au chantier #19 array-validation.*
- **`limesurvey:questionsLoaded` potentiellement mort** (`core/runtime.js:25`) : aucun code ne dispatch cet événement ; si le core l'émet via `jQuery.trigger()`, un listener DOM natif ne le reçoit pas → le bloc de ré-init all-in-one (`index.js:125-161`) serait du code mort.
- **`onPjax` sans retry** si jQuery absent à l'évaluation du bundle (`runtime.js:29-33`).
- **i18n incomplète** : « Ce champ est obligatoire », annonces SR du récap d'erreurs, ranking, captcha… en français codé en dur hors `core/i18n.js` (errors-dsfr.js:182,220,256,309 ; numeric-validation.js ; error-summary.js:168-179,291-298 ; conditional-aria.js ; ranking.js). RGAA 8.7/8.8 sur questionnaires non francophones.
- **`validation-messages.js:30-36` remplace les nœuds `vmsg_<qid>`** → références `aria-describedby` pendantes (described-by.js tourne avant) + mises à jour dynamiques EM perdues (« il reste N caractères ») ; `errors-dsfr.js:91` cherche un `.ls-em-tip` qui n'existe plus.
- **Listeners `{ once: true }` non ré-attachés** (`errors-dsfr.js:146-151`) : une question radio/checkbox qui retombe en erreur après une première correction ne se nettoie plus jamais.
- **~320 lignes mortes dans theme.js** (`initDsfrValidation` jamais appelée, l.996-1316) ; `enhanceFormValidation` (l.842-885) **double la validation captcha** avec deux messages contradictoires ; couplage caché : `combobox.js:18` dépend de la classe `dsfr-input` posée par theme.js:461.
- **6 `console.log` de debug** dans le bundle livré (`numeric-validation.js:421-447`, `sanitize.js:79,92`).
- Parsing regex « entre X et Y » fragile pour les bornes de somme (`numeric-validation.js:326,446`) : ni décimaux, ni négatifs, dépendant de la langue.
- Suppression globale des erreurs « bootstrap » (`theme.js:14-20`) qui masque de vraies régressions ; chemins d'assets en dur `/assets/packages/...` (theme.js:1336,1351) ; stubs Bootstrap dupliqués theme.js/bootstrap-stubs.js.

### CSS

- **824 `!important`** (custom.css 579, theme.css 240) ; course à la spécificité interne (`html body table…` custom.css:3235 pour battre ses propres `!important`) ; `all: unset !important` (custom.css:2989).
- **Duplication theme.css ↔ custom.css** : mêmes composants restylés avec valeurs contradictoires — `.question-container`, boutons bootstrap-buttons (×3 définitions, deux hovers contradictoires), `.alert`, `.modal` (display:flex theme.css:1089 vs display:none custom.css:873) → ~600 lignes mortes ou contradictoires.
- **Dark mode cassé localement** par hex en dur : radios matrices `#000091` + anneau `#FFF` (custom.css:1387-1398), en-têtes dual-scale `#161616` (theme.css:1883,1916,1946), `select` fond `#fff` (theme.css:2071), hovers `white` (theme.css:2172,2364).
- **`dsfr-grid-helpers.css` ~95 % superflu** : duplique des classes natives DSFR ; ses `--gutters-*` à base de `gap` entrent en conflit avec le système natif (padding/marges négatives).
- **Code mort** : bloc modales theme.css:1064-1224 (`.modal.dsfr-show` posée par aucun JS), ranking legacy (theme.css:2087-2129), `.survey-footer*`, `.skip-link`, `.progress*`, `.question-yesno` (la classe core est `yes-no`).
- **Breakpoints mélangés px/em** avec chevauchement à 768px exactement (max-width:768px ET min-width:768px tous deux actifs) ; `html{font-size:16px}` (theme.css:144) écrase la préférence utilisateur (DSFR : 100%) ; body < 16px sous 768px.
- DSFR embarqué : **v1.14.4, à jour** ✅.

### Iso-fonctionnalité types de questions

- **`repeat_headings` cassé (type F)** : `arrays/array/no_dropdown/answer.twig:24-30` force `doRender(answer_row)` au lieu de `{% include aRow.template %}` → les lignes repeat_header sont rendues comme des answer_rows corrompues.
- **Noms POST divergents du master** (cf. avertissement) : `{name}other` vs `_Cother` (listradio, list_dropdown), `_filecount` vs `_Cfilecount` (upload), `javatbd{name}Y/F` vs `_CY/_CF` (yesno/gender), aucun `javatbd…_C{value}` (5pointchoice).
- **Ranking** : `data-value="{{ansrow.code}}"` + `javatbd{{rankingName}}{{ansrow.code}}` vs core `data-value="{{ansrow.title}}"` + `javatbd…_S{sqid}` (`ranking/answer.twig:57-60`) → risque selects jamais remplis / array_filter d'items cassé. À tester en priorité.
- **Boilerplate (X)** : `{{ sTimer }}` et l'input caché supprimés → attributs `time_limit*` inopérants ; `{{ question|raw }}` potentiellement rendu en double.
- **`display_columns` ignoré** (listradio, plus de `ls-column-count-N`) ; hooks EM de ligne supprimés (id `javatbd`, `sDisplayStyle`, onclick de nettoyage du champ « autre »).
- **`label_input_columns` codé en dur** (multiplenumeric input, `fr-col-md-6` au lieu de `sLabelWidth`) ; **`answerwidth`/`columnswidth` supprimés** (array dropdown, plus de `<colgroup>`) ; **`placeholder` perdu** (longfreetext, hugefreetext) + `cols` (hugefreetext).
- **Variable `t` jamais définie dans les templates answer** : tous les `{{ t.xxx|default('…') }}` servent le fallback **français** (légendes, aria-labels SR) sur un questionnaire anglais.
- **file_upload entièrement réécrit** (XHR inline ~400 lignes au lieu de l'iframe + `UploadQuestionController` core) : fonctionnel mais surface de régression élevée à chaque évolution du endpoint core.
- **Classes d'erreur serveur retirées** (`ls-error-mandatory`, `has-error`) sur multiflexi/texts → marquage d'erreur 100 % dépendant du JS.
- **Fallbacks core non stylés DSFR** (rupture visuelle) : équation `*` (aucun template DSFR — contredit THEME_COVERAGE.md), list_with_comment en mode dropdown, multiflexi en checkbox layout, « autre » de multiplechoice_with_comments, carte (location_mapservice), date hour/minute, colonnes listradio.
- **`hideMultipleColumn` sans équivalent** : le no-op de theme.js:177-178 remplace les helpers core, la relevance des sous-questions est réimplémentée mais pas le masquage de colonnes de tableaux.

### Layouts, options, config.xml

- **Politique de confidentialité absente des pages token et inscription** (`showtokenpolicy`, `showregisterpolicy` non honorés ; vanilla token.twig:110, register_form.twig:123).
- **Inscription : attributs participants DD (dropdown) et DP (date) rendus en champ texte libre** (`register_form.twig:50-58`).
- **Opt-out : formulaire POST de confirmation (`optin_post_link`) absent** (`content/optout.twig:1-5`) → selon la version LS, le participant ne peut plus confirmer sa désinscription.
- **Options fantômes** : `showpopups`, `container`, `questionhelptextposition` consommées dans les twig mais commentées dans config.xml:62-65 (comportement figé, invisible dans l'admin) ; `show_footer_links` (config.xml:89) déclarée + documentée mais **jamais consommée**.
- **`css/icons-system.min.css`** présent dans le repo et cité par THEME_COVERAGE.md mais **jamais déclaré ni chargé**.
- **Chargement du JS DSFR par sniffing d'URL** (`theme.js:120-162`) : échoue en mode debug LS (asset manager off) → 404 silencieux sur `dsfr.module.min.js` (modales/header inertes).
- `showclearall` non testé sur le chemin d'affichage réel du bouton « Tout effacer » (`navigator_complement.twig:18`).
- `compatibility` ne déclare que `6.0` (le master vanilla déclare aussi 7.0) ; bloc `cssframework` `<name>dsfr</name>` mort (LS ne connaît que bootstrap).
- Pages `layout_survey_list/errors/maintenance` : n'étendent pas layout_global → dictionnaire `t.*` vide (aria-labels vides) + `data-fr-theme` absent.
- `main.twig:21` : commentaire Twig mal fermé (`}` au lieu de `#}`) — fonctionne par chance.
- Anti-bot : protection 100 % côté client (sessionStorage + display:none) — dissuasif seulement, à assumer dans la doc.

### Conformité DSFR

- **`fr-col-offset-2--lg` n'existe pas** (syntaxe : `fr-col-offset-lg-2`) → formulaires inscription / code d'accès / sauvegarde / chargement jamais centrés (register_form.twig:24,95 ; token.twig:8,21 ; save.twig:100 ; load.twig:89).
- **Gestion d'erreur des questions hors pattern DSFR** : `fr-fieldset--error` utilisé nulle part ; `fr-input-group--error` posé sur le conteneur de question (`question_container.twig:11`) et non sur le groupe du champ ; message obligatoire en FontAwesome + `text-danger` + `role=alert` statique (`mandatory_message.twig:8-14`).
- **Skiplink `#main-content` mort** hors pages de questions : seul `content/main.twig:26` pose `<main id="main-content">` — welcome, submit, register, save, load, optin/optout n'ont ni l'ancre ni de landmark `<main>` (RGAA 12.7).
- **Tableaux en markup fr-table legacy pré-1.10** : pas de `fr-table__wrapper/container/content`, pas de `<caption>` — sauf dualscale dont la caption sr-only est exemplaire, à généraliser. `role="radiogroup"` sur `<tr>` écrase la sémantique de ligne.
- **`data-fr-theme` posé côté serveur au lieu de `data-fr-scheme`** (`layout_global.twig:268`) : court-circuite le choix « système » de la modale fr-display.
- **Footer : « Accessibilité : totalement conforme » codé en dur** (`footer.twig:51`) + liens légaux en modales sans URL pérenne (la déclaration RGAA exige une page dédiée) ; date de déclaration `{{ "now"|date }}` qui change à chaque affichage (`modals/accessibility.twig:63`).
- `assessments.twig` (tables Bootstrap imbriquées) et `bootstrap_alert_modal.twig` restés Bootstrap ; gender/radio sans `fr-radio-group` ni fieldset ; language_changer en `fr-input` + classes Bootstrap 5 au lieu de `fr-select-group` ; checkbox clearall sans `fr-checkbox-group` avec labels imbriqués.
- **Classes `fr-*` inventées** à renommer avec un préfixe projet : `fr-progress`/`fr-progress__bar`, `fr-alert--dismissible`, `fr-form-group`, `fr-text--error`, `fr-input-wrap__prefix/suffix`, `fr-float-left`, `fr-range--vertical`, `fr-text-left/right`…
- **Icônes FontAwesome** (`fa-asterisk`, `fa-refresh`, `fa-exclamation-circle`) et **RemixIcon** (`ri-*`) résiduelles au lieu de `fr-icon-*`.
- **Inline massif** : ~420 lignes JS + 60 CSS dans `file_upload/answer.twig`, stubs dans `head.twig:26-140`, scripts inline dispersés — incompatible avec une CSP sans `unsafe-inline`.
- Stepper : `role="progressbar"` ajouté hors pattern (double annonce) ; aria-modal/aria-expanded écrits en dur dans le header (gérés par le JS DSFR).

## ✅ Points forts confirmés

- **Pipeline de build exemplaire** : `src/` modulaire, `build:theme:check` en CI, bundle livré strictement à jour (vérifié par rebuild + diff).
- **Documentation in-code remarquable** : chaque module et chaque hack explique le pourquoi, avec références RGAA et numéros d'issues.
- Idempotence systématique (flags `dataset.*`, namespaces jQuery), sécurité XSS soignée (textContent, DOMParser, whitelists).
- Header DSFR avec **composant traducteur `fr-translate`** et paramètres d'affichage `fr-display` conformes ; stepper conforme ; `<title>` dynamique « Étape X sur N » exemplaire (RGAA 8.5/8.6) ; relations ARIA des tableaux (headers/scope/aria-labelledby) très soignées.
- Couverture backoffice large : captcha (4 emplacements), assessments, quotas, save/load, opt-in, printanswers + export PDF, statistiques publiques, soft mandatory, mode pjax conservé.
- `token.twig` exemplaire sur le pattern d'erreur DSFR (le modèle à généraliser).
